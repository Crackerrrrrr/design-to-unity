# Design to Unity 设计方案

## 1. 项目定位

本项目优先提供完整设计信息与落地规则。本次目标聚焦 Unity：在保留 Design Implementation Packet 的同时，直接生成静态 UGUI prefab YAML、Sprite meta、source map，并通过 static verifier、Unity Editor validator 和 Unity MCP 实测形成闭环。

完整 Unity Editor API 正式 importer、Photoshop UXP 导出器生产化、Cocos / Godot / UE / Web writer 暂不纳入本次目标。

本项目的核心定位是：

> 从蓝湖设计稿、PSD / PSB 或 Photoshop / UXP 导出物中提取足够完整、结构化、低歧义的设计信息，并生成面向 Unity 静态 UI 落地的 Design Implementation Packet 与 prefab snapshot，让 AI 能够基于这些信息完成设计图到 Unity prefab 的可检查复原。

也就是说，本项目当前是一个“设计稿上下文 + Unity 静态 prefab 交付 MCP”，长期仍保留扩展到更多目标平台的结构。

长期目标平台可以包括：

- Unity
- Cocos Creator
- Godot
- Unreal Engine UMG
- Web 前端
- 后续可扩展到 Flutter、iOS、Android、React Native 等

## 2. 职责边界

### 2.1 本 MCP 负责

- 解析蓝湖链接。
- 获取蓝湖项目中的设计图列表。
- 获取指定设计图的 schema / sketch json / dds 数据。
- 下载图片、切图、图标等资源。
- 将蓝湖原始图层数据标准化为平台无关的 Design IR。
- 生成资源清单 Asset Manifest。
- 提供图层树、节点详情、样式信息、文本信息、坐标信息。
- 识别候选语义，比如按钮、进度条、滑块、可滑动区域、弹窗、列表、标题、图标、背景、装饰图。
- 根据目标平台返回落地规则 Handoff Profile。
- 给 AI 提供执行建议，让 AI 更好调用 Unity MCP 检查、实例化和继续加工 prefab。
- 在显式调用时，生成 Unity 静态 UGUI prefab YAML、Sprite meta 和资源复制结果。

### 2.2 本 MCP 不负责

- 不原地修改项目已有 Unity prefab；只生成新的静态 prefab snapshot 及配套 source map。
- 不直接创建 Unity scene。
- 不直接操作 Cocos 场景或节点。
- 不直接操作 Godot scene。
- 不直接创建 Unreal Widget Blueprint。
- 不直接写入业务代码。
- 不直接绑定项目已有脚本。
- 不直接手写 Godot tscn、Unreal uasset 等其他平台私有文件。
- 不把 direct YAML writer 用作业务脚本、动画、TMP 字体和项目自定义组件的生产绑定方案。

这些动作应该交给对应平台 MCP 或平台编辑器插件完成。

### 2.3 本次目标边界

本次验收只覆盖：

- PSD / PSB 与 Photoshop / UXP 导出物读取。
- Button、Slider / Progress、Toggle、Tab / ToggleGroup、TMP_InputField、TMP_Dropdown、ScrollRect / Scrollbar / RectMask2D 等 Unity UI 语义候选识别。
- Direct Unity prefab YAML、Sprite meta、source map 和资源复制。
- 静态 prefab verifier、Unity 侧 validator、Unity MCP 导入烟测。
- 为后续 Unity MCP 继续加工保留节点、资源、组件和来源映射。

本次不覆盖：

- Unity Editor API 正式 importer。
- Photoshop UXP 导出器生产化。
- Cocos、Godot、Unreal Engine、Web 的落地 writer。

## 3. 总体架构

```text
Lanhu / PSD / Future Source Adapter
  ↓
Raw Design Data
  ↓
Design Normalizer
  ↓
Design IR
  ↓
Asset Downloader / Asset Manifest
  ↓
Semantic Analyzer
  ↓
Design Implementation Packet
  ↓
Target Handoff Profile
  ↓
AI 调用目标平台 MCP 执行落地
```

### 3.1 Lanhu Source Adapter

负责与蓝湖接口交互，包括：

- 解析 URL 中的 `tid`、`pid`、`image_id`。
- 获取设计图列表。
- 获取分组信息。
- 获取设计图版本号。
- 获取 DDS schema。
- 获取原始 sketch json。
- 获取切图下载地址。

### 3.1.1 PSD Source Adapter

PSD Source Adapter 是蓝湖之后的下一类输入源，详细方案见 [PSD到Unity预制体扩展方案.md](PSD到Unity预制体扩展方案.md)。

第一阶段目标：

- 读取本地 PSD / PSB 文件。
- 遍历图层树和分组。
- 导出图层 PNG。
- 读取文本图层基础信息。
- 将 PSD 图层转换成与蓝湖同构的 Design Implementation Packet。
- 复用现有 Unity Handoff Profile 和 Direct Prefab YAML Writer。

PSD Source Adapter 不单独发明 Unity 落地规则，而是复用本项目已有的 packet、语义识别、资源清单和 Unity prefab YAML 写入能力。

### 3.2 Design Normalizer

将蓝湖不同格式的数据统一转换成平台无关结构。

需要兼容：

- Figma 格式
- Sketch 格式
- Photoshop 格式
- 蓝湖 DDS schema 格式

统一后，每个图层都应该具备：

- 唯一 ID
- 名称
- 类型
- 图层路径
- 坐标
- 尺寸
- 层级
- 可见性
- 样式
- 文本
- 图片资源引用
- 子节点

### 3.3 Asset Downloader

负责下载图片资源，并生成平台无关资源清单。

第一版建议只下载到本 MCP 的缓存目录，也可以支持用户传入目标工程资源目录。

资源清单中必须记录：

- 远程 URL
- 本地路径
- 原始尺寸
- 逻辑尺寸
- 格式
- 倍率
- 是否透明
- 建议命名
- 推荐导入设置

### 3.4 Semantic Analyzer

负责做轻量语义识别，但不做强制转换。

例如：

- 名称包含 `btn`、`button`、`按钮`，标记为 `button_candidate`。
- 大面积底图标记为 `background_candidate`。
- 位于顶部的大文本标记为 `title_candidate`。
- 多个结构相似的节点标记为 `list_item_candidate`。
- 遮罩 + 居中面板结构标记为 `dialog_candidate`。

语义识别只提供 hint，不强制 AI 或目标平台 MCP 执行绑定。

## 5. MCP 工具设计

### 5.1 设计图发现

```text
lanhu_design_list(url)
```

用途：

- 获取蓝湖项目中的设计图列表。
- 返回设计图 ID、名称、尺寸、分组、更新时间、序号。

返回重点：

- 设计图列表。
- 是否数量过多。
- 建议用户选择具体设计图。

### 5.2 准备设计交付包

```text
lanhu_design_prepare_packet(
  url,
  design_name_or_index,
  target = "unity|cocos|godot|unreal|web|generic",
  asset_output_dir = null,
  include_hidden = false,
  image_scale = "2x"
)
```

用途：

- 拉取指定设计图数据。
- 下载资源。
- 标准化节点树。
- 生成 Design Implementation Packet。
- 返回 `packet_id` 和摘要。

这是第一版最核心的工具。

### 5.3 获取完整 Packet

```text
lanhu_design_get_packet(packet_id)
```

用途：

- 返回完整 Design Implementation Packet。
- 适合节点较少的设计图。

如果节点太多，应返回提示，建议使用分页工具。

### 5.4 获取节点树

```text
lanhu_design_get_node_tree(
  packet_id,
  max_depth = 3,
  include_style = true,
  include_children = true
)
```

用途：

- 让 AI 分阶段读取设计结构。
- 避免一次返回过多 token。

### 5.5 获取节点详情

```text
lanhu_design_get_node_detail(
  packet_id,
  node_ids
)
```

用途：

- 查询指定节点的完整样式、文本、图片、语义、坐标。
- 适合 AI 对重点区域做精确落地。

### 5.6 获取资源清单

```text
lanhu_design_get_asset_manifest(packet_id)
```

用途：

- 返回所有资源路径、下载状态、尺寸、倍率、推荐导入设置。
- 供目标平台 MCP 下载、复制或导入资源。

### 5.7 获取平台交接规则

```text
lanhu_design_get_handoff_profile(
  packet_id,
  target
)
```

用途：

- 根据目标平台返回专门的坐标转换、组件映射、资源导入、文本映射、层级规则。

### 5.8 获取执行计划

```text
lanhu_design_get_implementation_plan(
  packet_id,
  target,
  mode = "static|componentized|adaptive"
)
```

用途：

- 给 AI 一份“应该如何调用目标平台 MCP”的建议计划。
- 不直接执行。

## 6. 平台 Handoff Profile

### 6.1 通用规则

所有目标平台默认遵循：

- 不使用远程 URL。
- 先导入资源，再创建节点。
- 保留设计图层级和视觉顺序。
- 保留原始设计尺寸、位置、颜色、字体信息。
- 对无法精确支持的效果生成 warning。
- 对疑似可交互组件只标记候选，不擅自绑定业务逻辑。
- 优先生成静态 View，再根据用户需求增强为组件。

### 6.2 Unity Profile

目标执行器：Unity MCP。

建议规则：

```text
- 使用 UGUI。
- 文本使用 TextMeshProUGUI。
- 图片使用 Image + Sprite。
- 根节点尺寸等于设计画布尺寸。
- 默认 Canvas 左上角坐标系。
- anchorMin = (0, 1)
- anchorMax = (0, 1)
- pivot = (0, 1)
- anchoredPosition = (x, -y)
- sizeDelta = (width, height)
- 图层按 Unity plan 的 create_nodes 顺序创建；同父节点内 z_index 方向由 source provider 决定，调用方不应脱离 create_nodes 自行重排。
- button_candidate 默认添加 Button 组件，但不绑定业务事件。
- progress_candidate / slider_candidate 默认添加 Slider 组件，fill / handle 子引用需要二次识别或人工确认。
- toggle_candidate 默认添加 Toggle 组件，状态图形和初始开关值在可推断时直接写入。
- tab_group_candidate / tab_candidate 默认添加 ToggleGroup + Toggle，默认选中项和 Toggle.group 在可推断时直接写入。
- dropdown_candidate 默认添加 TMP_Dropdown 组件，caption / template / item text 子引用在可推断时直接写入绑定。
- scroll_area_candidate 默认输出 ScrollRect 候选信息；viewport / content / item / scrollbar / handle 子角色在可推断时直接写入绑定，无法确定时再人工确认。
- list_item_candidate 可作为后续组件化依据，但第一版按静态节点创建。
```

组件映射：

```text
group -> GameObject + RectTransform
image -> GameObject + RectTransform + Image
text  -> GameObject + RectTransform + TextMeshProUGUI
shape -> GameObject + RectTransform + Image
button_candidate -> Image/TextMeshProUGUI + Button
progress_candidate -> Image + Slider
slider_candidate -> Image + Slider
toggle_candidate -> Image/TextMeshProUGUI + Toggle
tab_group_candidate -> ToggleGroup
tab_candidate -> Image/TextMeshProUGUI + Toggle
dropdown_candidate -> Image/TextMeshProUGUI + TMP_Dropdown
scroll_area_candidate -> ScrollRect candidate + Viewport + Content + Scrollbar
```

### 6.3 Cocos Creator Profile

目标执行器：Cocos MCP。

建议规则：

```text
- 使用 Node。
- 图片使用 Sprite。
- 文本使用 Label。
- 每个节点挂 UITransform。
- 根节点尺寸等于设计画布尺寸。
- 坐标需要从左上角转换到 Cocos 节点坐标。
- 默认 pivot 使用 (0, 1) 或根据 Cocos 版本转换 anchorPoint。
- SpriteFrame 使用下载后的本地资源。
```

组件映射：

```text
group -> Node + UITransform
image -> Node + UITransform + Sprite
text  -> Node + UITransform + Label
shape -> Node + UITransform + Sprite / Graphics
button_candidate -> Node + Button candidate
```

### 6.4 Godot Profile

目标执行器：Godot MCP。

建议规则：

```text
- 使用 Control 系 UI。
- 图片使用 TextureRect。
- 文本使用 Label 或 RichTextLabel。
- 按设计坐标设置 position 和 size。
- 根节点使用 Control。
- anchors 默认固定到左上。
- 对按钮候选使用 Button 或 TextureButton 由执行器按用户需求决定。
```

组件映射：

```text
group -> Control
image -> TextureRect
text  -> Label
shape -> ColorRect / TextureRect
button_candidate -> Button / TextureButton candidate
```

### 6.5 Unreal Engine UMG Profile

目标执行器：UE MCP。

建议规则：

```text
- 使用 UMG Widget。
- 根节点使用 Canvas Panel。
- 图片使用 Image。
- 文本使用 TextBlock。
- 坐标写入 CanvasPanelSlot position / size。
- anchors 默认左上。
- z_order 使用节点 z_index。
- button_candidate 可包装成 Button，但默认不绑定事件。
```

组件映射：

```text
group -> Canvas Panel / Overlay
image -> Image
text  -> TextBlock
shape -> Border / Image
button_candidate -> Button candidate
```

### 6.6 Web Frontend Profile

目标执行器：前端代码生成器或 Web MCP。

建议规则：

```text
- 第一版默认 absolute layout。
- 根容器 position: relative。
- 子节点 position: absolute。
- left = x px。
- top = y px。
- width / height 直接使用设计值。
- 图片使用本地 assets 路径。
- 文本保留 font-size、font-weight、color、line-height、text-align。
- 复杂布局后续可由 AI 基于语义转换为 flex/grid。
```

组件映射：

```text
group -> div
image -> img / div background-image
text  -> span / div
shape -> div
button_candidate -> button / div role=button candidate
```

## 7. AI 推荐工作流

### 7.1 静态页面落地

```text
1. 用户提供蓝湖链接。
2. AI 调用 lanhu_design_list。
3. AI 或用户选择设计图。
4. AI 调用 lanhu_design_prepare_packet。
5. AI 调用 lanhu_design_get_handoff_profile(target)。
6. AI 调用 lanhu_design_get_node_tree 了解结构。
7. AI 调用 lanhu_design_get_asset_manifest 获取资源。
8. AI 调用目标平台 MCP 创建页面、预制体、Widget 或组件。
9. AI 调用目标平台 MCP 截图。
10. AI 对照 Packet 中的节点和样式继续修正。
```

### 7.2 组件化落地

```text
1. 先按静态页面生成。
2. 读取 semantic_map。
3. 识别按钮、列表、弹窗、页签等候选结构。
4. 根据用户项目规范替换成已有组件。
5. 保留静态节点原始信息作为对照。
6. 生成组件化后的 diff 说明。
```

### 7.3 局部修改已有页面

```text
1. 用户指定蓝湖设计图和目标平台现有页面。
2. AI 获取 Packet。
3. AI 读取目标平台 MCP 的现有节点树。
4. AI 计算设计节点与现有节点的映射关系。
5. 只更新用户指定区域。
6. 保留业务绑定和人工修改。
```

## 8. 缓存和版本管理

### 8.1 缓存目录建议

```text
data/
  packets/
    {project_id}/
      {design_id}/
        {version_id}/
          packet.json
          node_tree.json
          asset_manifest.json
          semantic_map.json
          handoff_profiles/
            unity.json
            cocos.json
            godot.json
            unreal.json
            web.json
  assets/
    {project_id}/
      {design_id}/
        {version_id}/
          images/
          slices/
  raw/
    {project_id}/
      {design_id}/
        {version_id}/
          schema.json
          sketch.json
```

### 8.2 缓存策略

- 以蓝湖 `version_id` 作为强缓存键。
- 同一设计图同一版本重复请求时复用 packet 和资源。
- 版本变化时重新生成 packet。
- 资源下载失败时记录失败项，不让整个 packet 失败。
- 提供 force_refresh 参数强制重新拉取。

## 10. 第一阶段 MVP

第一阶段只做蓝湖输入和通用设计交付包，不接具体平台工程。

必须完成：

- `lanhu_design_list`
- `lanhu_design_prepare_packet`
- `lanhu_design_get_packet`
- `lanhu_design_get_node_tree`
- `lanhu_design_get_node_detail`
- `lanhu_design_get_asset_manifest`
- `lanhu_design_get_handoff_profile`
- Unity handoff profile
- Web handoff profile

第一阶段暂缓：

- Cocos / Godot / Unreal 复杂规则。
- 自动组件化。
- 复杂自适应布局。
- 双向 diff。
- 和目标平台 MCP 的自动联动。

## 11. 第二阶段能力

- 增加 Cocos / Godot / Unreal profile。
- 增加 semantic analyzer。
- 增加 list / dialog / button / tab / input 识别。
- 支持局部区域导出。
- 支持按节点下载资源。
- 支持平台执行计划生成。
- 支持 packet diff。
- 支持设计版本变更对比。

## 12. 第三阶段能力

- 支持 Figma、Sketch、本地 PSD 等更多 Source Adapter。
- 支持项目规范学习，例如命名规范、资源目录、组件库映射。
- 支持 Design Token 输出。
- 支持跨平台组件候选映射。
- 支持 AI 自动对照截图进行还原度评分。

## 13. 关键风险

### 13.1 蓝湖接口风险

蓝湖接口不是稳定公开 API，Web 端字段变化可能导致工具失效。

应对：

- 保留 raw schema。
- Normalizer 做多格式兼容。
- 错误时返回可诊断信息。

### 13.2 图层语义不稳定

设计师命名不统一，自动识别按钮、列表、弹窗可能误判。

应对：

- semantic_type 只作为候选。
- 不自动绑定业务逻辑。
- 将判断依据一并返回。

### 13.3 平台坐标差异

Unity、Cocos、Godot、UE、Web 坐标体系不同。

应对：

- Packet 中保存原始坐标。
- Handoff Profile 中单独描述转换规则。
- 不在原始 Design Node 上覆盖平台坐标。

### 13.4 复杂效果无法完整还原

渐变、阴影、混合模式、蒙版、模糊效果在不同平台支持不一致。

应对：

- 尽量提取原始参数。
- 在 warnings 中标注不确定效果。
- 必要时建议使用图片切图代替程序化还原。

## 14. 项目命名

```text
Design to Unity
```

CLI / MCP 配置推荐使用不含空格的键：

```text
DesignToUnity
```

Python 包发布名使用：

```text
design-to-unity
```

蓝湖只是其中一个 source adapter，后续 PSD / PSB / Figma / Sketch 都可以进入同一套设计实现数据包。

## 15. 总结

本项目当前的核心价值是让 AI 拿到足够完整的设计事实和 Unity 落地规则，并能直接得到可导入、可校验、可继续加工的静态 UGUI prefab。

本次目标内的协作模式是：

```text
Design to Unity 负责讲清楚设计是什么，并生成 Unity 静态 prefab snapshot。
Unity MCP 负责导入、实例化、检查组件引用、截图验证和继续加工。
AI 负责理解用户需求，并在 Design to Unity 和 Unity MCP 之间做决策和调度。
```

跨引擎 profile 仍是长期架构方向，但 Cocos、Godot、Unreal Engine、Web 前端 writer 不进入本次验收范围。
