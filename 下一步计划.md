## 2. 下一阶段目标

下一阶段重点不是继续扩平台，而是让 Unity MCP 更容易使用我们的输出，同时提供一个不依赖 Unity MCP 的静态 prefab YAML 快照出口。

目标是：

> 让 Design to Unity 从“能提供设计数据”升级为“能给 AI 和 Unity MCP 一份可执行的落地清单”，并在用户明确要求时直接生成 Unity 静态 prefab。

### 3.2 Unity 执行计划工具

已实现工具：

```text
lanhu_design_get_unity_plan(packet_id)
```

用途：

- 给 AI / Unity MCP 一份执行步骤。
- 明确资源导入顺序、节点创建顺序、组件类型和 warnings。
- 降低 AI 自己组织 packet 的成本。

当前返回内容：

```text
1. 导入 Sprite 资源。
2. 创建 ViewRoot。
3. 按 create_nodes 顺序创建节点；同父节点内的 z_index 方向由 source provider 决定，调用方不要自行重排。
4. group -> GameObject + RectTransform。
5. image -> GameObject + RectTransform + Image。
6. text -> GameObject + RectTransform + TextMeshProUGUI。
7. shape -> GameObject + RectTransform + Image 或项目自定义方案。
8. 对 button_candidate 只做候选标记，不自动绑定业务。
9. 写入 source metadata，方便重导入。
10. 保存 prefab。
11. 截图验证。
```

结构化返回字段：

```json
{
  "packet_id": "xxx",
  "target": "unity",
  "asset_imports": [],
  "root": {},
  "create_nodes": [],
  "semantic_candidates": {},
  "warnings": [],
  "rules": []
}
```
